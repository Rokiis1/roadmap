## What is k6?

**Explanation:**

k6 is an open-source load testing tool designed for testing the performance and reliability of APIs, microservices, and websites. It allows you to write test scripts in JavaScript, simulating virtual users that perform HTTP requests and other protocol operations.

<details>
    <summary>Overview:</summary>

- **Test Functions: Default vs. Named**
  - **Default Function:**
    - Defined with `export default function () { ... }`.
    - It serves as the primary entry point.
    - Use it when your test has a single, uniform flow or behavior.
  - **Named Functions:**
    - Defined with a specific name, `export function getRequest() { ... }`.
    - Useful for when you have multiple different test flows (one for load testing, another for stress testing).
    - Use these when you want to organize and run distinct behaviors within the same script.

- **Run the script:** `k6 run [filename].js`

</details>

## HTTP Requests

**Explanation:**

HTTP requests are the mechanism by which clients (like web browsers or performance testing tools) communicate with web servers. They are used to request data or to send data to a server.

<details>
    <summary>Overview:</summary>

- **Methods:** `GET`, `POST`, `PUT/PATCH`, `DELETE`

- **Headers:** Provide metadata such as content type, authentication tokens.

- **Body:** Contains data for methods that send data.

</details>

<details>
    <summary>Syntax:</summary>

  ```js
    // methods:  'GET', 'POST', 'PUT', 'PATCH', or 'DELETE'
    // url: Endpoint URL.
    // body: Request payload (null) if not needed.
    // params: Object containing options like headers.
    http.request(method, url, body, params);
  ```

</details>

<details>
    <summary>Examples:</summary>

1. **Example: GET**

    ```js
    import http from 'k6/http';

    export default function () {
      http.get('http://test.k6.io');
    }
    ```

2. **Example: GET with Authorization with Bearer Token or with Cookie**

    ```js
    import http from 'k6/http';

    export default function () {
      const url = 'http://test.k6.io';
      const params = {
          headers: {
              'Authorization': 'Bearer your_token_here', // Replace with actual token
          },
      };

      http.get(url, params);
    }
    ```

3. **Example: POST**

    ```js
    import http from 'k6/http';

    export default function () {
      const url = 'http://test.k6.io/endpoint';
      const payload = JSON.stringify({ key: 'value' });
      const headers = { 'Content-Type': 'application/json' };

      http.post(url, payload, { headers });
    }
    ```

4. **Example: PUT/PATCH/DELETE**

    ```js
    import http from 'k6/http';

    export default function () {
      const id = 12345; // Dynamic ID
      const baseUrl = `http://test.k6.io/endpoint/${id}`;
    
      const payload = JSON.stringify({ key: 'value' });
      const headers = { 'Content-Type': 'application/json' };

      // PUT request with ID in URL
      http.put(baseUrl, payload, { headers });
    
      // PATCH request with ID in URL
      http.patch(baseUrl, payload, { headers });
    
      // DELETE request with ID in URL (typically no payload is needed)
      http.del(baseUrl, null, { headers });
    }
    ```

</details>

## Metrics

**Explanation:**

Metrics are measurements that provide insights into the performance, reliability, and scalability of your system under test.

<details>
    <summary>Overview:</summary>

- **Standard built-in metrics:** Tese metrics are collected by k6 automatically during test execution to give you an overall picture of your test run.
  - **VU Iterations:** The number of times the default function is executed per virtual user.
  - **VUs (Virtual Users):** The number of active virtual users at various points during the test.
  - **Duration:** The elapsed time for which the test or individual iterations ran.
  - **Data Transfer:** Metrics such as bytes sent and received, which help gauge network usage.
  - **Checks:** While not a metric by themselves, the results of your checks (pass/fail) are aggregated into built-in metrics indicating percentage success.

- **HTTP-specific built-in metrics:** When you use HTTP requests in your test, k6 automatically tracks detailed metrics covering various aspects of the HTTP transaction.
  - **http_req_blocked:**
  - **http_req_connecting:** Time taken to establish the TCP connection to the server.
  - **http_req_duration:** Total time for an HTTP request, from start to finish (includes connecting, sending, waiting, and receiving).
  - **http_req_failed:** The rate of HTTP request failures (non-2xx responses).
  - **http_req_receiving:** Time taken to receive the response data from the server.
  - **http_req_sending:** Time taken to send the request data to the server.
  - **http_req_tls_handshaking:** Time for completing the TLS handshake (applicable for HTTPS requests).
  - **http_req_waiting:** Time spent waiting for the server to send the first byte of the response (the server’s processing time).

</details>

## Checks

**Explanation:**

Checks provide a way to validate the behavior and correctness of responses during your tests. They act as assertions that determine whether certain conditions are met after an HTTP request or any other operation.

<details>
    <summary>Overview:</summary>

- **Purpose:** To verify that responses meet the expected criteria (correct status codes, response times, or data formats).

- **Usage in k6:**
  - Implemented using the `check()` function in k6 scripts.
  - You can specify multiple conditions in a single `check()` call, which makes your tests more concise.

- **Benefits:**
  - Immediate feedback if something goes wrong.
  - Easier debugging and error handling through detailed reporting.

</details>

<details>
    <summary>Syntax:</summary>
  
```js
import { check } from 'k6';

check(response, {
    'condition label': (r) => r.property === expectedValue,
    // Multiple conditions can be added here
});
```

</details>

<details>
    <summary>Examples:</summary>

  ```js
  export default function () {
      const response = http.get('https://example.com');

      check(response, {
          'status is 200': (r) => r.status === 200,
          'response time is less than 500ms': (r) => r.timings.duration < 500,
      });
  }
  ```

</details>

## Sleep

**Explanation:**

The `sleep()` function in k6 is used to pause execution for a specified number of seconds. This delay simulates "think time" between user actions.

<details>
    <summary>Examples:</summary>

```js
import http from 'k6/http';
import { sleep } from 'k6';

export default function () {
    http.get('https://example.com/api');

    sleep(1);
}
```

</details>

## Options

**Explanation:**

Options in k6 allow you to configure the behavior of your test runs. They let you define parameters like virtual user load, test duration, thresholds, scenarios. By setting options, you can simulate various load patterns and assert performance requirements.

<details>
    <summary>Overview:</summary>

- **Load Configuration:** Define the number of virtual users and how they ramp up or down over time using stages or scenarios.
  - **VUs:** The number of virtual users (VUs)
  - **Stages:** Allow you to define a timeline for how many VUs are active. Each stage specifies a duration and a target number of VUs.
  - **Ramp up or down:** This refers to gradually increasing (ramp-up) or decreasing (ramp-down) the number of VUs over a specified period
  - **Scenarios:** Scenarios provide an advanced way to structure your load tests. They let you define multiple, concurrent load patterns within the same test script.

- **Thresholds:** Set performance goals for metrics (response time, error rate). If thresholds are not met, tests will be flagged as failed.
  - **Common Thresholds for APIs:**
    - `'http_req_failed': ['rate<0.01']`: This sets an acceptable error rate, ensuring almost all requests succeed. Less than 1% of requests should return an error.
    - `'http_req_duration': ['p(95)<200']`: This means that the 95th percentile of all request durations must be under 200ms. 95% of requests should have a response time below 200ms.
    - `'http_req_duration': ['p(99)<400']`: This stricter metric ensures that nearly all requests finish within an acceptable range. 99% of requests should have a response time below 400ms.

- **Scenarios:** Advanced way to structure your load tests. Instead of using a single VU configuration (like constant VUs or stages), scenarios let you define multiple, concurrent load patterns that can simulate different aspects of user behavior in one script.
  - **Executor:** Determines the strategy for simulating virtual users.
    - **constant-vus:** Maintains a fixed number of VUs for a specified duration.
    - **ramping-vus:** Gradually increases or decreases the number of VUs over defined stages.
  - **Scheduling Parameters:** Depending on the executor.
    - **vus:** The number of virtual users to run (for constant scenarios).
    - **duration:** Total duration for which the scenario should run.
    - **stages:** An array of objects, each containing a duration and a target number of VUs for ramping scenarios.
    - **startVUs:** Specifies the initial number of virtual users when the test begins (used with certain executors, like `ramping-vus`).
    - **startTime:** Delays the start of a scenario relative to the overall test start time.
  - **Exec:** Specifies the name of the function to execute for that scenario.

</details>

<details>
    <summary>Examples:</summary>
  
1. **Example: Options 1 Constant VUs Configuration**

    ```js
    export let options = {
        vus: 10,         // Constant 10 virtual users
        duration: '1m',  // Test runs for 1 minute
    };
    ```

2. **Example: Options 2 Stages-Based Load Configuration**

    ```js
    export let options = {
      stages: [
          { duration: '30s', target: 5 },  // Ramp-up to 5 VUs over 30 seconds
          { duration: '1m', target: 10 },  // Ramp-up to 10 VUs over 1 minute
          { duration: '30s', target: 0 },  // Ramp-down to 0 VUs over 30 seconds
        ],
    };

3. **Example: Options 3 Combined Stages and Thresholds Configuration**

    ```js
    export let options = {
      stages: [
          { duration: '30s', target: 5 },   // Ramp-up to 5 VUs over 30 seconds
          { duration: '1m', target: 10 },   // Ramp-up to 10 VUs over 1 minute
          { duration: '30s', target: 0 },   // Ramp-down to 0 VUs over 30 seconds
      ],
      thresholds: {
          // 95% of requests must finish within 500ms
          'http_req_duration': ['p(95)<500'],
          // Less than 1% of requests should fail
          'http_req_failed': ['rate<0.01']
        },
    };

4. **Example: Options 4**

    ```js
    export let options = {
      scenarios: {
          myScenario: {
              executor: 'constant-vus', // startegy simulating virtual users
              vus: 10,          // 10 virtual users continuously
              duration: '1m',   // Test runs for 1 minute
              exec: 'scenarioTest', // Function to execute
        },
      },
    };

    export function scenarioTest() {
      http.get('https://example.com/api/endpoint');
      sleep(1);
    }
    ```

</details>

## Built-in Virtual User Variables

**Explanation:**

K6 provides built-in global variables that help create dynamic, unique data for each virtual user and iteration

- `__VU`: Represents the unique identifier of the current Virtual User. Each virtual user in your test run receives its own `__VU` value, making it useful for generating unique data or debugging.

- `__ITER`: Indicates the current iteration number for the virtual user. It increments each time the default function is executed, allowing you to track and differentiate execution cycles.

## Test lifecycle

**Explanation:**

The test lifecycle in k6 governs the sequence of events from the start of a test to its completion. Understanding this lifecycle helps in setting up scripts correctly and ensuring resources are properly managed.

<details>
    <summary>Overview:</summary>

1. **Initialization Phase:**
    - Code outside of functions is executed once when the script loads.
    - Useful for importing modules or initializing data needed during the test.

2. **Setup Phase:**
    - If a `setup()` function is defined, it runs before the virtual users (VUs) start executing the main test function.
    - This phase is useful for preparing test data or performing initial configurations.

3. **Execution Phase:**
    - The `default` function is the entry point for each VU and is executed repeatedly according to the specified load configuration.

4. **Teardown Phase:**
    - If a `teardown()` function is defined, it runs once after all VUs finish their iterations.
    - Use this phase to perform cleanup tasks, such as releasing resources or aggregating results.

</details>

## Scenarios

**Explanation:**

Scenarios in k6 enable you to define multiple, distinct load test cases within one script. Each scenario can simulate a different user behavior or account for various load patterns (such as constant, ramping, or arrival rate). This approach is especially useful when you want to test multiple aspects of your application concurrently.

<details>
    <summary>Overview:</summary>

- **Multiple Load Patterns:** Define different scenarios with specific virtual user (VU) counts, durations, and other properties.

- **Customization:** Configure each scenario independently, allowing for diverse testing approaches (stress testing versus soak testing).

- **Flexibility:** Run multiple scenarios concurrently, giving a comprehensive view of how different parts of your system respond under load.

</details>

## Secret source

**Explanation:**

Secret sources in k6 refer to strategies for managing sensitive information such as API keys, tokens, or passwords—required during your load tests without hardcoding them in your scripts. By separating secrets from your code, you reduce security risks and prevent accidental exposure in version control systems or logs.

## Test authoring

**Explanation:**

Test authoring in k6 involves writing modular, maintainable JavaScript code that simulates user behavior under various load patterns. It leverages k6's built-in functions, lifecycle hooks, and modules to design tests suited for your application's performance requirements.

<details>
    <summary>Overview:</summary>

- **Structure:**
  - **Initialization:** Import modules and set up global variables.
  - **Setup/Teardown:** Optional hooks for pre-test configuration and cleanup.
  - **Default Function:** Contains the core test logic executed by each virtual user (VU).

- **Components:**
  - **Checks & Thresholds:** Validate response conditions and enforce performance parameters.
  - **Scenarios:** Define different load patterns and test profiles in one script.
  - **Modules:** Organize reusable helper functions and utilities.

</details>
