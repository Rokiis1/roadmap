## Summary Fundamentals

Fundamentals of testing define **what testing is, why it is important, and how quality is understood in software development**.

Testing is the process of **evaluating a system to verify that it works as expected and meets user needs**. It supports quality by identifying issues and reducing risk.

Two key concepts define correctness

- **Verification** ensures the system is built correctly according to specifications  
- **Validation** ensures the system solves the right problem and delivers value  

Testing is not limited to execution. It includes **planning, monitoring and control activities** that ensure testing is effective and aligned with project goals.

Testing can be

- **Dynamic** which involves executing the software  
- **Static** which involves reviewing artifacts without execution  

Testing has clear objectives

- **Evaluate work products** to detect issues early  
- **Find defects** by triggering failures  
- Ensure **coverage of critical areas**  
- **Reduce risk** in the system  
- Verify that the system meets **requirements and expectations**  

Testing and debugging are related but different

- **Testing identifies defects**  
- **Debugging analyzes and fixes them**  

Key terminology defines problems in software

- **Error** is a human mistake  
- **Defect** is a flaw in the system  
- **Failure** is incorrect behavior during execution  
- **Root cause** is the underlying reason for a problem  

Testing follows important principles

- **Testing shows the presence of defects**, not their absence  
- **Exhaustive testing is impossible**, so prioritization is required  
- **Early testing** reduces cost and risk  
- **Defect clustering** (Pareto principle) means most defects are found in a small number of areas  
- **Pesticide paradox** means repeating the same tests reduces effectiveness over time  
- **Testing is context dependent**, meaning approaches vary by project  
- **Absence-of-errors fallacy** means a defect-free system may still fail to deliver value  

The **test process** provides structure

- **Planning** defines objectives, scope and approach  
- **Monitoring and control** track progress and adjust actions  
- **Analysis** identifies what to test  
- **Design** creates test cases and data  
- **Implementation** prepares environments and procedures  
- **Execution** runs tests and reports defects  
- **Completion** finalizes testing and captures lessons learned  

Testing work produces outputs

- **Artifacts** such as test cases, scripts and data  
- **Testware** which includes all reusable testing materials  
- **Deliverables** such as test plans and summary reports  

Testing involves different roles

- **Test management** focuses on planning, coordination and control  
- **Test engineers** perform analysis, design, execution and reporting  
- Responsibilities may vary depending on team structure  

Quality is managed through two perspectives

- **Quality Assurance (QA)** focuses on preventing defects through process improvement  
- **Quality Control (QC)** focuses on detecting defects in the product  

Testing independence improves objectivity

- Testing can be performed by developers, peers or independent teams  
- Independence helps uncover issues but should be balanced with collaboration  

Modern development promotes the **whole team approach**

- Quality is a **shared responsibility**  
- Developers, testers and stakeholders collaborate continuously  
- Communication and knowledge sharing improve overall quality  

Key takeaway

- Testing is a **continuous, structured and collaborative activity**  
- It focuses on **risk reduction, defect detection and value delivery**  
- Quality depends on both **process (QA)** and **product (QC)**  
- Effective testing combines **principles, process and teamwork**
