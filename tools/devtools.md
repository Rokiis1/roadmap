Web applications are built and run inside the browser, which means understanding browser behavior is essential for development.

Modern browsers provide built-in tools that allow developers to inspect, debug and analyze how a web application works in real time.

These tools make it possible to view the structure of a page, monitor network requests, store, inspect data and analyze performance.

You can think of these tools as a control panel for the browser, giving you visibility into what is happening behind the scenes.

This set of tools is commonly known as **DevTools**.

To start exploring how DevTools can be used in practice, we begin with a feature that helps simulate different devices and screen sizes.

## Device toggler

The **Device toggler** allows you to simulate different screen sizes and devices directly in the browser.

This is useful when testing how a website behaves across mobile, tablet and desktop environments.

You can switch between predefined devices or define custom screen dimensions to match specific scenarios.

This helps verify that layouts adjust correctly and that the application remains usable on different screen sizes and orientations.

It is commonly used when working with **responsive design** to ensure consistent user experience across devices.

Once we understand how an application behaves across different devices and how it communicates with the server, the next step is to inspect how the page is actually structured and rendered in the browser.

For this, we use the **Elements Panel**.

## Elements Panel

The **Elements Panel** allows you to inspect and modify the structure of a web page.

It displays the **HTML** of the page as a tree, showing how elements are nested and organized.

This helps you understand how the page is built and how different parts are connected.

When you select an element, the panel highlights it on the page.

This makes it easier to identify where a specific part of the layout is coming from.

You can also view and edit the **CSS styles** applied to an element.

This allows you to test changes directly in the browser, such as modifying colors, spacing, or layout.

These changes are temporary and only affect the current session.

The panel also shows additional information such as **computed styles, box model, and applied rules**.

This helps you understand how styles are calculated and how layout properties like margin, padding, and size affect the element.

The Elements Panel is commonly used for **debugging layout issues, inspecting structure, and testing style changes** without modifying the source code.

It is one of the most frequently used tools when working with the frontend of a web application.

While the Elements Panel helps inspect structure and styles, it is also important to observe how the application behaves during execution and to debug issues in real time.

For this, we use the **Console**.

## Console

The **Console** allows you to interact with a web application in real time.

It is primarily used for **debugging and inspecting behavior** while the application is running.

You can view messages such as **logs, warnings, and errors** generated by JavaScript code.

These messages help identify issues and understand what is happening during execution.

The Console also allows you to run JavaScript directly in the browser.

This makes it possible to test code, inspect variables, and experiment with functionality without modifying the source files.

For example, you can check values, call functions, or interact with elements on the page.

Errors shown in the Console often include additional details such as file names and line numbers.

This helps locate the source of a problem more quickly.

The Console is commonly used to **debug issues**, **test small pieces of code** and **observe application behavior** during development.

It is an essential tool when working with JavaScript and dynamic web applications.

While the Console helps observe and debug application behavior, it is also important to inspect how data is stored and managed within the browser.

For this, we use the **Application Panel**.

## Network Panel

The **Network Panel** allows you to monitor all requests made by a web application.

It shows how resources such as **HTML, CSS, JavaScript, images, and API calls** are loaded and how the browser communicates with the server.

Each request includes details such as the **method, URL, status code, and response time**.

This helps identify issues such as **failed requests, slow responses, or unexpected behavior**.

The panel also provides a **timeline view** that shows when each request starts and how long it takes to complete.

This helps understand the **loading sequence** and detect resources that delay the page.

When inspecting a request, you can view **headers, payload, and response data**.

This is useful for debugging **API calls** and verifying that correct data is being sent and received.

You can also observe different phases of a request, such as time spent **waiting, connecting to the server, and downloading content**.

These stages help explain where delays occur during loading.

The Network Panel also allows you to simulate different **network conditions**.

For example, you can test how the application behaves on **slower connections** to identify performance issues.

Overall, this panel is essential for **debugging**, **performance analysis** and **understanding how an application communicates with the server**.

While the Network Panel shows how data is exchanged with the server, it is also important to understand how data is stored and managed inside the browser.

For this, we use the **Application Panel**.

## Application Panel

The **Application Panel** allows you to inspect and manage data stored by a web application in the browser.

It provides access to different types of client-side storage such as **cookies**, **localStorage** and **sessionStorage**.

This makes it possible to see what data is stored, how it is structured, and how it changes during application usage.

You can view cookies associated with a domain, including their **name**, **value**, **expiration** and **security attributes**.

This is useful for understanding how authentication or session data is managed.

The panel also allows you to inspect **localStorage** and **sessionStorage**, where applications store `key:value` data on the client side.

You can read, modify or remove stored values directly from the browser.

This helps when testing application behavior or debugging issues related to stored data.

In addition to storage, the Application Panel may include other browser data such as **cache**, **indexed databases** and **service workers**, depending on the application.

The Application Panel is commonly used for **debugging storage-related issues**, **inspecting authentication data** and **verifying how an application persists information in the browser**.

It is especially important when working with features that rely on client-side stat.

While the Application Panel focuses on stored data, it is also important to understand how the application performs during loading and interaction.

To analyze performance and identify slow operations, we use the **Performance Panel**.

## Performance Panel

The **Performance Panel** allows you to analyze how a web application behaves over time.

It records activity in the browser and shows how different parts of the application contribute to loading and interaction performance.

When you start a recording, the browser captures events such as **JavaScript execution**, **rendering**, **layout** and **painting**.

This helps you understand what happens during page load and user interactions.

The panel displays a timeline that shows how these activities are distributed over time.

This makes it easier to identify slow operations or tasks that block the main thread.

You can inspect long-running tasks and see which parts of the code are responsible for delays.

This is useful for detecting performance issues caused by heavy JavaScript or inefficient rendering.

The Performance Panel also shows how frames are rendered, which helps identify problems such as **lag**, **dropped frames** or **slow animations**.

By analyzing these details, you can understand why an application feels slow and where improvements are needed.

The Performance Panel is commonly used for **performance analysis**, **identifying bottlenecks** and **optimizing user experience**.

It is especially helpful when working on applications that require smooth interactions and fast response times.
